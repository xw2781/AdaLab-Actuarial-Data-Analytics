ArcRho Plan

DFM RPC Bridge Sync Request System

Add a DFM-level sync workflow that sends Details page context to the data-engine through request files, compares local and remote method JSON files, and presents one contextual action based on which version is newer.

Version v0.4 Last updated 2026-05-10 Timeout 8 seconds Canonical DFM JSON keys Name-only local DFM files

Feature Summary

Entry Point

Add a Sync button in dfmPathBar.

Primary Request

Send Function = DFM with Details fields plus DataPath.

Follow-up Request

Keep Using Local does not send Function = SyncDFM; it keeps local JSON and removes the remote RPC JSON.

Filename standard: DFM filenames must use @<origin_length>@<development_length>. Existing local helpers should be updated to this order, with no migration fallback for old files.

User Workflow

1

Open DFM and click Sync

The action starts from the DFM tab path bar.

2

Handle unsaved edits

If the tab is dirty, ask whether to save and proceed. Cancel stops the sync.

3

Send DFM request

Backend writes Function = DFM request under requests\RPC bridge.

4

Wait and compare

Backend waits for remote JSON, compares JSON last modified timestamps, and returns local/remote metadata.

5

Select Old or New

The draggable and resizable floating window resizes the comparison cards with the window, lets the Notes preview expand from 42px up to 300px, shows Local in a framed top-left label on the left card and Remote Server in a framed top-left label on the right card, hides JSON file paths, adds a NEW seal on the newer card, color-codes wide ratio-pattern cells with dark grey for common exclusions, green for newly excluded cells on the new card, and red for removed exclusions on the old card, shows card-specific legends for green/red states, shows Ratios-table origin/development labels on cell hover, and highlights notes deletions in red and additions in green. After a version action is selected, the comparison window closes and a smaller same-style DFM Sync message box shows waiting/final status.

Selected version is Local

Result: Keep Using Local

Keep local JSON unchanged and remove the returned remote RPC JSON. No Function = SyncDFM request is sent.

Selected version is Remote

Result: Update Local DFM

Overwrite the local JSON with the remote JSON, then reload ratios, averages, and notes into the current DFM tab.

Request Contracts

Local DFM method JSON files use DFM@<full_reserving_class_path>@<DFM Name>.json. Within one project/reserving-class pair, Name is the sole local DFM instance identity, so the same Name cannot represent multiple period lengths or input triangles. Origin Length and Development Length live inside the local JSON as origin length and development length. RPC bridge response/status files remain length-qualified because the remote data-engine request contract expects the requested lengths in DataPath.

Initial DFM Request

The initial request includes all Details page fields plus the expected returned method JSON path.

Function = DFM
ProjectName = <project_name>
Path = <full_reserving_class_path>
MethodName = <DFM Name>
OutputVector = <output_vector>
InputTriangle = <input_triangle>
OriginLength = <origin_length>
DevelopmentLength = <development_length>
DecimalPlaces = <decimal_places>
DataPath = <expected_remote_json_path>
UserName = <current_user>

Initial DataPath

<server_root_path>\projects\<project_name>\methods\RPC bridge\DFM@<full_reserving_class_path>@<DFM Name>@<origin_length>@<development_length>.json

SyncDFM Request

Function = SyncDFM is retained for explicit remote-update workflows. Keep Using Local does not send this request.

Function = SyncDFM
ProjectName = <project_name>
Path = <full_reserving_class_path>
MethodName = <DFM Name>
OutputVector = <output_vector>
InputTriangle = <input_triangle>
OriginLength = <origin_length>
DevelopmentLength = <development_length>
DecimalPlaces = <decimal_places>
DataPath = <expected_sync_status_json_path>
UserName = <current_user>

SyncDFM DataPath

<server_root_path>\projects\<project_name>\methods\RPC bridge\SyncDFM@<full_reserving_class_path>@<DFM Name>@<origin_length>@<development_length>.json

Returned JSON Contracts

Minimum DFM Method JSON

{
  "ratio pattern": [[1, 0], [1, 1]],
  "average formulas": ["Volume - all"],
  "average index": [[1, 1]],
  "notes": "...",
  "last modified": "2026-05-10T12:00:00.000Z"
}

SyncDFM Status JSON

{
  "ok": true,
  "status": "passed",
  "message": "Remote DFM status updated."
}

Canonical Method Payload

{
  "ratio pattern": [],
  "average formulas": [],
  "average index": [],
  "summary rows": [],
  "summary order": [],
  "ultimate vector": [],
  "notes": "",
  "name": "",
  "output type": "",
  "input triangle": "",
  "origin length": 12,
  "development length": 12,
  "decimal places": 4,
  "ultimate ratio decimal places": 2,
  "ratio basis dataset": "",
  "last modified": ""
}
Use only the canonical DFM method JSON keys shown above. summary rows is the persisted average formula row list; summary hidden is not part of the contract. Old files and alias spellings are intentionally out of scope.

Backend Design

File Responsibility
frontend/app_server/schemas/dfm_rpc_bridge.py Pydantic schemas and validation for project, reserving class, method name, lengths, and timeout.
frontend/app_server/services/dfm_rpc_bridge_service.py Path building, request writing, wait loops, JSON last modified compare, local overwrite, keep-local cleanup, optional SyncDFM update, and RPC JSON deletion.
frontend/app_server/api/dfm_rpc_bridge_router.py Thin router exposing sync, compare, apply, keep-local, and update-remote routes.
Router registration point Include the new router without moving business logic into the API layer.

Routes

Route Behavior
POST /dfm/rpc-bridge/sync Write Function = DFM, wait for remote method JSON, return comparison metadata.
POST /dfm/rpc-bridge/compare Compare existing local and remote JSON last modified timestamps without sending a request.
POST /dfm/rpc-bridge/apply Copy remote JSON over local JSON after Update Local DFM, return payload for reload.
POST /dfm/rpc-bridge/keep-local Keep local JSON unchanged and remove the returned remote RPC JSON without writing a request file.
POST /dfm/rpc-bridge/update-remote Write Function = SyncDFM, wait for status JSON, return pass/fail and message.

dfm_rpc_bridge_dialog.js

Create and wire the Sync button inside dfmPathBar.

Timestamp Logic

Use the canonical last modified property inside each DFM JSON file. Local DFM JSON writes this value on each GUI save event. Filesystem modified times may still be returned as secondary metadata, but they do not decide which version is latest.

State Condition UI Behavior
remote_latest Remote JSON last modified is newer than local JSON last modified. Show Use Remote Version.
local_latest Local JSON last modified is newer than remote JSON last modified. Show Keep Using Local.
same_time JSON last modified timestamps are equal within tolerance. Show Local and remote are already in sync. No primary action.
remote_missing Local exists, remote is missing. Tell the user remote DFM JSON is missing. No primary action.
local_missing Remote exists, local missing. Error state. This should not occur because the app saves before proceeding.
both_missing Neither file exists. Error state. This should not occur after save-and-proceed.

Validation and Error Handling

Backend Statuses

  • 400: missing Details page fields.
  • 404: project folder not found.
  • 408 or status = "timeout": expected JSON did not appear.
  • 409: invalid apply state or filename collision.
  • 423: local or remote file locked.
  • 500: invalid JSON or unexpected filesystem failure.

Frontend Rules

  • Disable Sync while one sync is running.
  • Show elapsed waiting state.
  • Preserve dirty-state semantics.
  • Save-and-proceed prompt appears before request creation.
  • Reload data only after backend confirms write success.
