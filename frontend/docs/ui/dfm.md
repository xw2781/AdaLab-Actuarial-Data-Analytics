# Frontend: Dfm

## Purpose
<!-- MANUAL:BEGIN -->
DFM feature (details/ratios/results/notes) on top of dataset context, including method metadata in Details.
DFM Notes tab now reuses the shared `notes_editor_interactions.js` behavior layer (dataset-style path highlighting with a right-click file menu, `Tab`/`Shift+Tab` indentation, `Esc` to exit editing, spellcheck off, top whole-note display formatting toolbar for font family/size/color and bold/italic/underline/strikethrough) while preserving DFM's existing persistence/save flow in `dfm_persistence.js`; file paths render as deep-blue underlined non-bold text, right-clicking a rendered path shows `Open File` and `Copy File Path`, while focused the editor shows plain text only (rich/path-highlight layer hidden), toolbar interactions do not end editing mode, and rich rendering returns after focus leaves both the textarea and toolbar.
<!-- MANUAL:END -->

## External Interfaces
<!-- MANUAL:BEGIN -->
- Exchanges `arcrho:*` messages with shell and workflow iframe.
- Reuses dataset APIs and reserving class selectors.
- DFM Data grid reuses shared dataset `#tableWrap` frame/border rendering (native square `border` stroke with no outline/pseudo overlay to keep corners stable and avoid border paint overlapping grid cells after table render at non-integer zoom/scaling).
- Shared dataset/DFM table keeps native smooth scrolling inside `#tableWrap`; after scroll idle, viewport position auto-snaps to nearest row/column grid boundaries to reduce sticky-header partial-overlap artifacts from sub-cell offsets.
- DFM Ratios tab now uses a dedicated framed scroll host (`#ratioWrapHost` -> `#ratioWrap`) so sticky headers/first-column overlays are contained within the ratio grid viewport instead of page-level scrolling.
- DFM Data tab now wraps `#tableWrap` inside `#tableWrapHost`, so shared dataset sticky header/first-column behavior remains pinned inside the table viewport during Data-tab scrolling.
- DFM page boot waits for the shared Dataset boot to finish so project-specific origin/development labels are refreshed before Data and Ratios tabs render their triangle headers.
- Opening a DFM tab now lands directly on the Details page instead of showing a startup chooser. The Details `Name` field has a `...` selector that reads the current project's cached `dfm_method_index.json` and lists only method names associated with the selected Reserving Class path. Selecting a name commits the normal Details lookup, so the existing local DFM JSON load flow applies the method settings.
- DFM Results tab table now shows per-origin columns in this order: `Latest <Input Triangle Name>` (latest observed value from DFM Data-tab triangle row), `Reserve` (`Ultimate - Latest`), and `Ultimate`; Results header cells allow wrapped text so long input triangle names are readable instead of truncating to one line.
- DFM Results tab adds a `Ratio Basis` selector above the table (typed input + `...` dataset picker window backed by shared `dataset_name_picker.js`). The picker/datalist now expose all dataset names from current-project `dataset_types` (including calculated and non-triangle/vector types), while Result-column extraction currently supports only `Triangle`/`Vector` selections; unsupported types show a clear error status. Results also includes a dedicated `Ultimate Ratio Decimals` input (default `2`) used only for the `Ultimate Ratio` percent column formatting. After selection, Results appends `Ratio Basis: <dataset>` and `Ultimate Ratio` columns after `Ultimate`; the ratio-basis column uses values from the selected dataset at the current DFM `Origin Length` basis (triangle basis datasets contribute the latest diagonal value per row), and `Ultimate Ratio = Ultimate / Ratio Basis`. Results also appends a `Total` row that sums numeric columns; `Ultimate Ratio` total is computed as `Total Ultimate / Total Ratio Basis`.
- DFM method save writes Results CSV under `projects/<project>/data/<ReservingClassFolder>/<DatasetName>.csv`; `ReservingClassFolder` is the single caret-sanitized reserving-class path folder, and `DatasetName` uses Details `Name` unless an aggregated output name is supplied. When source period is Monthly/Quarterly, save also emits aggregated variants (Monthly -> Quarterly/Half-Year/Year, Quarterly -> Half-Year/Year) in the same reserving-class data folder.
- DFM path bar includes a `Sync` button for the RPC bridge workflow. Clicking Sync prompts the user to save first when the current DFM tab is dirty, sends a backend `Function = DFM` request containing Details page fields plus `DataPath`, waits up to 8 seconds for the returned remote DFM JSON under `data/tmp/<ReservingClassFolder>`, then shows a taller draggable, resizable `Review Changes & Select One Version` window with local/remote `last modified` timestamps from the JSON plus `ArcRho - Local`/`RPC - Remote` cards without showing JSON file paths. The comparison body has its own vertical scrollbar so the bottom of both diff cards remains reachable when content is taller than the window; wide snapshot grids and long notes still scroll inside their own preview boxes. Resizing the window also resizes the comparison cards, with snapshot content scrolling inside the cards when space is tight and the Notes preview expanding vertically from its 42px minimum height up to 300px when extra space is available, then scrolling inside the notes box instead of truncating long notes. The local version always appears on the left and the remote version always appears on the right, with the source label in the card's top-left corner framed by a dark blue border and light blue fill. The newer card displays a `NEW` seal and green-framed timestamp; the older card displays a red-framed timestamp. Each version card snapshots the excluded ratio triangle as a masked triangle with rectangular cells that compact for large matrices; cells with value `2` outside the inferred active diagonal are masked, while value-`2` cells inside the active diagonal are shown with a light yellow fill for missing data. The active diagonal is inferred from the latest row containing data and that row's last non-`2` cell. Commonly excluded ratio cells are dark grey, exclusions added in the new version are green only on the new card, and exclusions removed from the old version are red only on the old card. The green `Newly excluded` legend appears only on the new card, the red `No longer excluded` legend appears only on the old card, and a yellow `Missing inside triangle` legend appears only when those cells exist. Hovering a ratio cell shows the same origin/development labels used by the Ratios triangle table. The Average Formulas section renders a compact formula-row/development-column selection diff instead of a plain formula-name list, using dark grey for commonly selected cells, green only on the new card for newly selected cells, and red only on the old card for no-longer-selected cells. Notes snapshots compare the full `notes` text, highlight deleted old-version text with a red background, and highlight newly added new-version text with a green background. If the local version is newer, selecting the Local card shows a `Confirm` primary action; after one more app-styled RPC server write confirmation dialog, the backend sends `Function = SyncDFM` with the local method JSON path and waits for the bridge status result. Selecting `Keep Using Local` in non-write-back cases keeps the local JSON unchanged and deletes the remote RPC JSON without sending `Function = SyncDFM`; selecting the remote version shows a `Confirm` primary action, overwrites the local method JSON using the same row-compact DFM method JSON formatter as normal saves, then reloads the current DFM tab. If applying the remote version changes Input Triangle, Origin Length, or Development Length, DFM refreshes the dataset-dependent UI and immediately saves the recalculated local method JSON; if that final save fails, the DFM tab is marked dirty so the user can save before closing. After a version action is selected, the comparison window closes and a smaller same-style DFM Sync message box shows waiting status while the action is running; successful final messages become non-blocking notifications that disappear after 2 seconds, while warnings/errors stay visible and can close with `Esc` or by clicking outside. Equal `last modified` timestamps show `Local and remote are already in sync` with no primary action, and missing remote JSON is informational only.
- When a DFM method payload is applied from local lookup, RPC remote-to-local sync, template load, or ArcBot JSON edit and the payload changes Input Triangle / Origin Length / Development Length, the embedded dataset layer reruns with the new Details inputs and refreshes header labels so Data/Ratios reflect the new JSON immediately without closing and reopening the DFM tab.
- DFM template save is triggered from shell `File -> Save as Template` (relabel of `Save As...`) only when DFM `Details` tab is active; the Details page button `Save as Template` is removed.
- DFM Details page is organized into two framed sections to match the app style: `Object Path` (`Project`, `Reserving Class`, `Name`) and `Method Inputs & Settings` (`Output Vector`, `Input Triangle`, and remaining DFM settings such as lengths/decimal places). The `Load Settings From Template` action is placed at the bottom-right of the Details page (outside the framed sections). The Details form content also uses added top spacing.
- Workflow-embedded DFM objects can bind their Project and Reserving Class inputs to Global Control defaults, displaying `Default Project (<project>)` and `Default Path (<path>)` while DFM lookups, saves, RPC bridge calls, and output/vector pickers use the resolved values.
- New standalone DFM tabs read the last selected Project Name from `%APPDATA%\ArcRho\local_project_prefs.json` when the Project field is otherwise blank, then load that project's shared `lastReservingClassPath` plus `dfmObject` Details defaults from `projects/<project>/users/<windows-login>/preferences.json` when the corresponding Details fields are blank. Selecting a project in the project tree picker updates the same local last-project preference used by Dataset Viewer immediately, records it in the last-three `recentProjectNames` list, and reloads the selected project's DFM defaults; changing the Project input box itself only applies project-scoped DFM defaults. Changing the DFM Reserving Class field updates the shared project-user `lastReservingClassPath`.
- Shared dataset/DFM table shows light square arrow buttons with minimal SVG icons at scrollbar ends in the Data grid viewport; each click advances one grid unit (one row or one data column), with disabled states at scroll boundaries.
- Shared dataset/DFM table renderer always displays all `values` columns; when development header labels are fewer than data columns, remaining header cells are intentionally blank rather than truncating tail data.
- Shared dataset/DFM table uses a single thin grid divider between the last data row and `Total` footer row (avoids doubled/thick separator rendering), and when `Total` is hidden the last body row keeps its bottom grid border so the bottom grid remains visible.
- Shared dataset/DFM table hides the `Total` footer row when current Dataset Type `Formula` (from `dataset_types`) contains `*` or `/`.
- Shared dataset input layer validates selected Dataset Type dependency resolvability (project `dataset_types` formulas + direct Dataset entries in `field_mapping`) before generation requests; when dependency resolution fails, it performs an ArcRhoTri-style `/arcrho/tri/precheck` local CSV existence check and allows refresh only if that local CSV is already present, otherwise unresolved prerequisites still trigger a blocking popup/status message. In this fallback path, clear-cache refresh is blocked and execution is limited to local-CSV-only load attempts.
- Shared dataset input/status layer does not append Dataset Type `Formula` to app status text; after successful dataset render, status is `Ready`.
- Shared Dataset Type picker `Picker Preferences` (`Double Click to Select`, `Close Window after Selection`) are stored in the current project-user preference file. Shared reserving-class picker preferences and hidden paths are stored there as well, so duplicated projects carry the same per-user picker defaults.
- Reserving Class tree picker highlights only one active node at a time: the deepest visible node on the active path (if deeper levels are collapsed, highlight falls back to the nearest visible ancestor), and shows the selected path text in the picker title (fallback title remains `Reserving Class` when no path is selected).
- Shared reserving-class path normalization preserves literal `/` characters inside a class name; only `\` is treated as the path segment delimiter.
- Reads `/dataset_types?project_name=...` to drive the Details `Output Vector` picker (filtered to `Data Format = Vector`, including calculated datasets).
- Reads `/dataset_types?project_name=...` to drive the `Input Triangle` picker (filtered to `Data Format = Triangle`, including calculated datasets).
- DFM `Output Vector` and `Input Triangle` `...` buttons now reuse the shared `dataset_name_picker.js` window picker with DFM-specific dataset-format filters (`Vector`/`Triangle`) while including calculated datasets; the existing typed dropdown behavior remains available on the inputs themselves and now matches the same inclusion rule. The shared picker also includes header-level `Name Search`, `Data Format`, `Category`, and `Calculated` filter popups to narrow large dataset lists without changing DFM-specific save/load behavior. In the shared picker, long `Name` values wrap within the first column instead of truncating, header column widths reserve space for sort/filter controls so labels remain readable, sort direction indicators use filled SVG triangles to match Project Settings table semantics, internal column boundaries can be drag-resized from the header grid lines (the dragged left column changes width while the last column auto-compensates to keep total table width constant), the `Calculated` column renders disabled checkboxes in rows (while its filter options remain `Yes` / `No`), titlebar tools include a category-group collapse/expand toggle plus `Clear all filters` (left of picker preferences), the picker opens at an initial height of about two-thirds of the host app viewport height, and the picker window's resize max bounds are refreshed when the host app window is resized so enlarging the app window also expands the picker's available resize range (vertical resize max is based on host-window height, not the picker's current top position). During dragging, the picker may move beyond the host window's left/right/bottom edges (top edge remains constrained so the title bar stays reachable). `Data Format`/`Category`/`Calculated` filter popups use `(All)` as the explicit no-filter state (default = only `(All)` checked), clicking a specific option clears `(All)`, clicking `(All)` clears specific checks, and right-clicking an option row inverts all non-`(All)` selections.

## Data/State/Caches
<!-- MANUAL:BEGIN -->
- Persists ratio selection/templates via DFM persistence modules.
- Tracks dirty flags and active DFM tab state.
- Caches eligible output-type names per project for the Details picker and enforces selection-only input.
- Selecting a Details `Output Vector` auto-synchronizes the Details `Name` field to the same string.
- Details `Output Vector` supports type-to-filter dropdown search; empty input shows all available output-vector options.
- During async project-scoped `Output Vector` option refresh/load (for example immediately after page refresh with prefilled Project), the picker does not clear the field while the `Output Vector` input is actively focused/being typed in.
- Reuses shared dataset input boot logic (`dataset_main.js`) and server project user preferences for project-specific shared `lastReservingClassPath` / Dataset defaults when no query/workflow values override the inputs.
- For new DFM pages, `Name`, `Output Vector`, and `Input Triangle` start empty instead of prefilled defaults.
- DFM method JSON identity/save naming uses `projects/<project>/data/<ReservingClassFolder>/DFM@<Name>.json`; within a project/reserving-class pair, the DFM `Name` is the sole instance identity and must be unique. A single Name cannot represent multiple DFM instances with different period lengths or input triangles. The project is represented by the containing folder, and the reserving class is represented by the containing caret-sanitized folder, not repeated in the JSON filename.
- DFM no longer creates, reads, updates, or deletes `dfm_method_names.json` or the browser-local cached DFM Name key; local method lookup uses only the current Details fields and the standard local DFM JSON path.
- DFM method JSON save/load persists the selected Results `Ratio Basis` dataset name (`ratio basis dataset`) so reopening the same DFM method restores the Ratio Basis selection automatically.
- DFM method JSON also persists Details `Name` (`name`), `Output Vector` (`output type`), `Input Triangle` (`input triangle`), Origin Length (`origin length`), and Development Length (`development length`) and restores them on local method load; users may manually change these Details inputs afterward and the changes follow normal DFM dirty/save behavior.
- DFM Origin Length and Development Length are independent method arguments. The hidden shared dataset length-link control is off on DFM pages. When Origin Length changes, Development Length is automatically adjusted only if the current pair breaks the required relationship (`Origin Length >= Development Length` and `Origin Length = Development Length * integer`); editing Development Length directly does not trigger an automatic rewrite.
- DFM suppresses centered loading popups from both DFM method setting loads and the embedded Dataset Viewer loader; dataset refresh and reserving-class validation should report through normal status UI without blocking the DFM page with a modal loading window.
- DFM Details `Decimal Places` (default `4`) controls ratio number formatting across the Ratios tab display (main ratio grid, summary/selected rows, and ratio chart labels/tooltips) and is persisted in DFM method JSON as `decimal places` so reopening the method restores the same Ratios formatting.
- DFM method JSON also persists the Results tab `Ultimate Ratio Decimals` control as `ultimate ratio decimal places` (default `2`) so reopening the method restores `Ultimate Ratio` percentage formatting independently of Ratios-page decimal formatting.
- DFM method JSON now uses a canonical GUI-tab grouped shape (`json format = arcrho-dfm-method-by-tab-v1`) with `details tab`, `data tab`, `ratios tab`, `results tab`, `notes tab`, and `method metadata` sections. The grouped JSON persists the ratio selection matrix under `ratios tab`.`ratio triangle`.`excluded`, ratio-tooltip labels, Results output, GUI save time, and rounded 4-decimal read-analysis snapshots for input data triangle values, ratio values, average formula values, plus the input triangle CSV path; Data-tab `development labels` use raw Data headers such as `2m`, while Ratios-tab ratio-triangle `development labels` use GUI-display ratio headers such as `(1) 2-14`. Analysis arrays trim trailing `null` cells from each row while preserving internal `null` cells, and `excluded` rows are clipped to the corresponding `ratio values` row length so non-value columns such as the trailing Ult mask column are not persisted. These snapshots are ignored when restoring editable method selections. The DFM method JSON formatter writes any 2D array as one compact child row per JSON line, including extra or nested 2D arrays. ArcBot active-context temp JSON uses the same grouped method shape and host-applied edits remain grouped. See `docs/ui/dfm_json_format.md` for the current DFM JSON format notes.
- DFM method/template JSON persists average formula settings under a columnar `average formulas` object with `label`, `custom average formula settings`, `selected`, and `values`; `summary rows`, `summary hidden`, and `summary order` are not used. `label` is the persistent formula identity and runtime-only `id` values are generated after load. The average formula table order follows the `average formulas.label` sequence, and dragging rows updates the actual formula metadata so the UI order and saved formula order stay synchronized. The average formula table delete action is disabled when only one row remains, so the table always keeps at least one row. The selected-summary area shows `Selected`, `Cumulative`, and `% Developed`; `% Developed` is calculated as `1 / Cumulative` for each development period.
- Right-clicking anywhere in the DFM Ratios selected-summary table opens a selected-summary context menu with `Show Percentage Developed Curve`. The floating `% Developed Curve` window reads the rendered `% Developed` row and places actual data points at the matching Data-tab development-month labels for each ratio column, so a ratio period like `(1) 11-23` plots at month 11. The curve starts at month 1 and `0%`, then interpolates monthly values to the first actual point. Each displayed ratio column owns the segment ending at that column's actual point; the `Curve Type By Segment` section presents those segments in a horizontally scrollable two-row table with `Period` labels and in-cell `Curve Type` dropdowns. Segment curve choices default to linear interpolation and include smooth, step, exponential, logarithmic, and power choices. The window also has an optional Prior Project input with the shared project tree picker; selecting a prior project does not change the active DFM project, and clicking `Add Curve` fetches the same Reserving Class path and DFM method name from that project and overlays its `% Developed` curve. If the matching DFM method cannot be found or cannot produce curve points, the window shows the error and leaves the current graph unchanged. DFM method JSON persists the curve chart `x-axis label` and each segment's selected curve type/display name under `ratios tab`.`percent developed curve`, and method load restores those curve choices for the next time the window opens.
- The DFM Ratios average formula table right-click menu includes `Plot Table Data`, which opens a draggable and resizable floating plot window. The plot reads the current rendered summary table values, charts each average formula row plus the current `Selected` row across the current development-period columns, and shows a row legend. The plot window toolbar includes zoom, hand-drag, and pencil tools; zoom mode changes the chart cursor to zoom-in, dragging a chart region zooms into that selected area, a single chart click resets to the original full view, hand-drag mode pans the full chart view with pointer-captured dragging, pencil mode lets users click an average-formula point to make that formula the selected value for the clicked development period, and `Esc` deselects the active tool. Pencil clicks ignore the `Selected` line itself. The plot legend follows the Dataset Viewer chart legend behavior: clicking a legend label row or checkbox hides/shows that line, `Select All` controls all lines, individual toggles keep at least one line visible, and right-clicking a legend row isolates that line or restores all lines when it is already isolated.
- When loading DFM JSON from RPC/template/local method payloads, DFM reconstructs runtime average formula rows from the `average formulas` object. Supported inferred labels are `Volume - all`, `Volume - <n>`, `Simple - all`, `Simple - <n>`, and the same labels with `Ex hi/lo` or `Ex hi/lo x<n>` suffixes; unresolved labels are skipped.
- DFM template files use `.arc-dfm` extension only (`DFM_Template@<Project>@<ReservingClass>.arc-dfm`); legacy `.arcdfm` suffix is not accepted.
- After required Details inputs are all populated, DFM checks whether the instance JSON exists at the standard local save path; if missing, it does not show the former inline warning/`Confirm` action beside `Name` (new-object guidance is status-bar only).
- On the same local instance-file lookup, once DFM Details has committed values for `Project`, `Reserving Class`, and `Name`, DFM automatically checks for a matching local method JSON. If found, it auto-loads the file and reports the loaded file path in the status bar. If not found, it reports a yellow status-bar warning (via shell status `tone=warn`) with the exact text `This method object has not been created yet, changes will be saved to a new container.` The lookup does not wait for `Input Triangle` / `Output Vector` / period lengths.
- Details `Input Triangle`, `Origin Length`, and `Development Length` are saved properties of the current DFM instance, not local instance lookup keys; changing them marks the current DFM dirty and follows the normal dataset refresh behavior without reloading method JSON from disk. Saving the DFM object writes the adjusted length values back into the stored method JSON.
- DFM Details `Name` is a plain text input with a project/path-scoped selector backed by `dfm_method_index.json`; it does not use the older browser-local saved-name dropdown or delete action.
- Temporary debug aid: DFM local-method lookup currently emits short status-bar debug messages (for trigger/skip/checking-path) prefixed with `Debug: DFM local method lookup ...` to help verify the frontend lookup flow during Details input changes.
- DFM local-method lookup scheduling is debounced and reason-coalesced; higher-priority `details-change` lookups are not downgraded by later `tab-activated` triggers (avoids false `dirty + tab-activated` skips immediately after Details edits).
- Shell tab activation/state-request messages refresh DFM edit-state reporting only; they do not trigger local method lookup.
- Existence checks run on committed input changes (`change`/selection/blur`) rather than every typing keystroke.
- DFM responds to shell `arcrho:assistant-context-request` messages with the active DFM tab, Details field values, dirty state, standard method JSON path, and current in-memory method payload so ArcBot can analyze the active method even when host-side direct reads from the server share are denied. After an ArcBot JSON edit is applied by the host, DFM receives `arcrho:assistant-json-updated` and schedules a local method reload.
- DFM watches the active local method JSON file for external disk changes. When the page is clean, a changed JSON file is reloaded into the active GUI automatically; changed Ratios exclusion cells and Average Formula selection cells briefly flash blue so API/ArcBot edits are easy to inspect. When the page is dirty, DFM does not overwrite in-memory edits and posts a warning so the user can save or reload deliberately.

## Common Change Tasks
<!-- MANUAL:BEGIN -->
1. Add a DFM tab capability: update orchestrator + tab module.
2. Modify ratio/result behavior: sync `dfm_ratios_tab.js` and `dfm_results_tab.js`.
3. Adjust Details output-type rules: update filter logic in `dfm_details.js` and keep workflow DFM settings compatibility.

## Known Risks
<!-- MANUAL:BEGIN -->
- Cross-tab sync is message-driven and easy to desynchronize.
- Persistence schema changes can break saved templates.
- If dataset type definitions drift from expected columns, Details output-type options may appear empty.
<!-- MANUAL:END -->
